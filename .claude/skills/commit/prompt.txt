When invoked with /commit, follow this flow:

## Step 1: Run the strict build
- Run: mkdocs build --strict
- If it fails, stop and report the strict-mode failures in detail
- Do NOT proceed if the build fails
- This repo has no test suite; the strict build is the gate

## Step 2: Review Changes
- Run: git diff --staged (to see staged changes)
- Run: git status (to see working tree)
- Understand what changed and why

## Step 3: Determine Type and Scope
From the actual changes, derive:
- Type: feat | fix | chore | ci | docs | build | refactor | test | perf | revert
- Scope: guide | api | blog | nav | agents (pick most relevant)

Examples:
- Adding a user-guide page → docs(guide)
- Correcting wrong documented behavior → fix(guide)
- Updating the API reference / openapi.json → docs(api)
- Adding a release blog post → docs(blog)
- Editing mkdocs.yml nav → docs(nav) or build(nav)
- Updating .claude/ machinery → chore(agents)

## Step 4: Write Commit
- Subject: <type>(<scope>): description
- Rules:
  - ≤72 characters
  - lowercase
  - no period at end
  - imperative mood ("add" not "added")
- Body: only if "why" is non-obvious
- Include Co-Authored-By footer

## Step 5: Execute
- Create and execute the commit
- Verify with git log

## Important
- MUST have mkdocs build --strict passing before commit
- MUST use imperative mood
- MUST match Conventional Commits format (valid types only)
